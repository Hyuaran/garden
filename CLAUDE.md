## プロジェクト概要

- **Gardenシリーズ**: FileMaker・kintoneの代替となる自社Webアプリケーション
- **技術スタック**: Next.js (App Router) / Supabase / Vercel / TypeScript / Tailwind CSS

## 11. 横断調整セッション運用（2026-04-23 追加）

### 役割分担

Gardenシリーズは複数のClaudeセッションが並行稼働するため、以下の役割分担で運用する：

| セッション | 役割 |
|---|---|
| **a-main（横断調整セッション）** | 全モジュール横断の指示・ルール改訂・セッション間の整合性調整・ユーザーからの全体方針伝達 |
| **a-soil / a-root / a-tree / a-leaf / a-bud / a-bloom / a-seed / a-forest / a-rill** | 担当モジュールの実装作業 |
| **b-main** | 東海林Bアカウントでの作業セッション（A使用枠到達時の引き継ぎ先） |

### 横断指示の受け取り方

他セッションから **「【横断セッション(a-main)からの共有】」** で始まるメッセージを受け取ったら：

1. **指示内容を最優先で確認**する（現在の作業の区切りで一度手を止める）
2. 影響を受けるモジュール・アクション要否を判断
3. 必要なアクションを実施
4. **疑問点があれば横断セッション経由でユーザーに確認**を依頼する（他モジュール実装セッション同士で勝手に合意しない）

### 恒久ルール vs 一時指示

| 種別 | 伝達経路 | 反映タイミング |
|---|---|---|
| **恒久ルール** | CLAUDE.md（親/子）・MEMORY.md に直接記載 | 次回各セッション起動時に自動反映 |
| **一時指示** | 「【横断セッション(a-main)からの共有】」テンプレート形式でユーザーが各セッションに手動コピペ | その場で即時適用（セッション内のみ） |

### 横断調整セッションからの発信判断基準

横断調整セッション側（a-main）は以下の基準で発信方式を決める：

- モジュール仕様の正式変更・新ルール追加・恒久的な合意事項 → **CLAUDE.md / MEMORY.md を直接編集**
- 当日の作業優先順位・一時注意・他セッションへの確認依頼 → **テンプレート形式でユーザーに渡す**

## 12. 作業工程の実績蓄積ルール（2026-04-23 追加）

### 目的

横断調整セッション（a-main）が、ユーザーから突発的に「〇〇モジュールの進捗は？」「全体の遅延はどれくらい？」と聞かれた時に**即答できる**ようにするため、各セッションは作業実績を共通ファイルに蓄積する。

### 蓄積先ファイル

`docs/effort-tracking.md`（リポジトリ直下の `docs/` 配下）
※ 既存の MEMORY.md `feedback_effort_tracking` を踏襲

### 記録タイミング

以下の区切りで随時追記する：
- 新機能・新フェーズ着手時（予定時間を記録）
- 機能・フェーズ完了時（実績時間・差分を記録）
- 方針変更・見積もり修正時
- 日次作業終了時（未完了タスクの中間記録）

### 記録フォーマット

`docs/effort-tracking.md` に以下の表形式で追記：

| 日付 | モジュール | タスク | 予定時間 | 実績時間 | 差分(±h) | 理由 | 担当セッション |
|---|---|---|---|---|---|---|---|
| 2026-04-23 | Garden-Tree | Phase A 認証画面実装 | 4h | 6h | +2h | RLS設計で詰まった | a-tree |
| 2026-04-23 | Garden-Forest | v9 レビュー修正 | 2h | 1.5h | -0.5h | 軽微修正のみで済んだ | a-forest |

### 運用ルール

- 作業区切りで**各セッションが追記 → `git commit & push`**
- A/B セッション共通（東海林A・B どちらでも同じファイルに追記）
- 追記したら軽量な commit メッセージでOK（例：`docs: effort-tracking 更新 (tree)`）
- セッション名（a-tree / a-forest / b-main 等）は末尾列に必ず明記

### 突発報告への対応（a-main の責務）

ユーザーから「〇〇の進捗は？」「全体の遅延は？」等の確認があった際、a-main は以下の順で即答する：

1. `git pull` で `docs/effort-tracking.md` を最新化
2. 該当モジュール・期間の行を抽出
3. **予定 / 実績 / 差分（±h） / 遅延理由** を表形式で報告
4. 必要なら全体サマリ（合計予定 vs 合計実績）も併記

### 初回導入時の対応

このルールが新規追加された時点で、各セッションは以下を実施：
- 現在進行中のタスクがあれば、**過去分を遡って記録**（把握できる範囲でOK）
- 未来分（直近予定タスク）を**予定時間付きで先行記入**

## 13. 自律実行モード（2026-04-24 改訂 v2 / 初版: 2026-04-23）

### 🔒 専用セッション方式（核心）

自律実行モードは **a-auto セッション専用** で発動する。a-auto 以外のセッション（a-main / a-forest / a-tree 等）では発動しない。

### 目的

ユーザー（東海林さん）が**確認作業できない時間帯**（就寝中・会議中・外出中・集中別作業中）に、5時間使用枠を有効活用して作業を進め、復帰時にレポートとして結果を受け取る運用モード。

### セッション役割分担

| セッション | 自律実行モード | 役割 |
|---|---|---|
| a-main（横断調整） | 発動しない | ユーザー対話・朝の判断・**周知配布支援** |
| **a-auto（自律実行専用）** | **★ここのみで発動** | 夜間・会議中の自律作業全般 |
| a-soil / a-root / a-tree / a-leaf / a-bud / a-bloom / a-seed / a-forest / a-rill | 発動しない | インタラクティブ作業専用（**周知受信役も兼ねる**） |
| b-main | 発動しない | 東海林Bバックアップ |

### 想定シーン

| シーン | 想定時間 | 補足 |
|---|---|---|
| 就寝前 | 5〜8時間 | 5時間使用枠をフル活用。朝レビュー |
| 会議中 | 30分〜2時間 | 会議前に発動、終了後に確認 |
| 外出中 | 1〜4時間 | 移動・接客などで確認できない時間 |
| 集中別作業中 | 任意 | 別タスクに集中している間、並行処理 |

### 発動方法（ユーザー操作）

**必ず a-auto セッション**に対し、次のいずれかを入力：

- 標準形：「`自律実行モード発動してください`」
- シーン明示形：「`自律実行モード発動してください。就寝前`」 / 「`会議中（約〇時間）`」 / 「`外出中（約〇時間）`」
- 対象モジュール指定形：「`自律実行モード発動してください。対象: a-tree, a-forest`」
- 優先指定形：「`自律実行モード発動してください。優先度: 〇〇`」
- タスク明示形：「`自律実行モード発動してください。タスク: 1)〇〇 2)△△`」

a-auto 以外のセッションに同じ発動指示が来た場合、「自律実行モードは a-auto 専用です。a-auto 側で発動してください」と応答して停止。

### 🔓 ユーザー不在時の許可モード設定（2026-04-24 追記）

就寝中・会議中・外出中など**ユーザーが画面を見られない時間帯**に発動する場合、以下を**発動前に必ず**実施：

1. a-auto セッションの入力欄で **Shift+Tab** を押す
2. 画面下に「⏵⏵ accept edits on」や「accept all edits」等の表示が出るまでサイクル
3. このモードでは**すべての許可確認が自動承認**される
4. その状態で自律実行モード発動プロンプトを投入

### 安全性の担保

Shift+Tab モード使用時でも、以下は依然として保護される：

- `.claude/settings.json` の **deny ルール**は最優先で適用
  - `rm` / `rmdir` / `Remove-Item` / `del`（削除系）
  - `git reset --hard` / `git push --force` / `git push -f` / `git clean -f`（Git破壊系）
  - `git branch -D`（ブランチ強制削除）
  - `git push origin main` / `git push origin develop`（保護ブランチへの直接push）
- Shift+Tab は**許可確認をスキップするだけ**で、deny を無効化しない
- つまり、対策A（settings.json 拡張）＋対策B（Shift+Tab）の**二重保険**で安全運用可能

### セッション単位の有効範囲

- Shift+Tab モードは**現セッション限り**
- セッション再起動後は通常モード（プロンプトあり）に戻る
- ユーザーが対面で作業する際は通常モード（発動時点でShift+Tabを押さない or 再起動）

### a-auto の動作フロー

1. 発動を受けて、対象モジュールまたはコンテキストを把握
2. 対象モジュールに該当するブランチに `git checkout`（または a-auto 作業専用ブランチ `feature/xxx-yyy-auto` を新規作成）
3. 他モジュールセッションが同ブランチで作業中でないか確認（直近コミット時刻・handoff 記述から推定）
4. 作業を実行（§13の制約内）
5. commit + push
6. 次の対象モジュールへ移動、同様の処理
7. 全タスク完了または停止条件で終了
8. **復帰時レポート + 周知メッセージ + 全体サマリ**を生成（Phase 1 で詳細規定）

### 他モジュールセッションとの協調

a-auto が作業するとき、**対象モジュールセッションは待機**が原則（ブランチ競合防止）。

- 発動時：ユーザーが対象モジュールセッションに「a-auto で自律実行中、このセッションは待機」と伝える
- 完了後：a-auto → a-main 経由で Phase 2 の短文配布 → Phase 3 の5ステップで復旧

補助策：**a-auto は可能な限り自作業専用ブランチ（`feature/xxx-yyy-auto`）を作成**することで、モジュール側の既存ブランチと衝突しない運用を推奨。

### 自動コンテキスト把握

a-auto は発動時、対象モジュールごとに以下を読み込む：

1. 対象モジュールに対応するブランチの直近コミット（`git log --oneline -20`）
2. `docs/handoff-*.md` / `docs/wip-*.md` の最新ファイル
3. `docs/effort-tracking.md` 内の対象モジュール担当分
4. 自セッション名＝`a-auto`（複数モジュールまたぎ前提）

### タスクリスト自動生成の優先順

1. ユーザーが発動時に明示した優先タスク・対象（最優先）
2. 対象モジュールの `docs/wip-*.md` / `docs/handoff-*.md` の「次にやるべきこと」
3. `docs/effort-tracking.md` で遅延（差分 +h）のあるタスク
4. 直近コミットから推測される論理的次手
5. 上記で具体化できない場合 → **停止**

### 判断ルール（迷ったらこうする）

- 設計方針の判断が必要になったら → **即時停止**、`docs/pause-YYYYMMDDHHMM-a-auto.md` 作成
- 既存コードへの変更：動作を変えない範囲のみOK、動作変更は設計判断扱い → 停止
- テストが通らない → 失敗理由を記録して次のタスクへ
- エラー解決に3回以上失敗 → 該当タスクを飛ばす
- ユーザーへの確認事項が発生 → 即停止
- 仕様解釈に幅あり → 既存実装・既存ドキュメントに最も近い選択

### 禁止事項

- ファイル・レコード削除（`.claude/settings.json` の deny でシステム的にも禁止済み）
- `main` / `develop` ブランチへの直接 push
- Supabase 本番データへの書き込み（書き込みが必要なタスクは停止）
- kintone のレコード追加・編集・削除
- 新規 npm パッケージのインストール
- 勝手な仕様変更・設計変更
- ユーザー事前承認なしの外部API課金呼び出し
- 他モジュールセッションが作業中のブランチへの同時push
- CLAUDE.md / MEMORY.md の編集（恒久ルール変更は判断事項）

### コミット方針

- 対象モジュールの作業ブランチ（`feature/xxx-yyy-auto` 等）で commit
- メッセージに **[a-auto]** タグを含める（例：`test(bud): [a-auto] 振込CRUDテスト追加`）
- push は作業の区切りで（最低でも復帰前に1回）
- 作業実績は対象モジュールフォルダの `docs/effort-tracking.md` に追記（§12準拠）

### 停止条件

1. 初期タスクリスト全件完了
2. 5時間使用枠が**90%に到達**
3. 設計判断・ユーザー確認事項が発生

7. 結果を `01_東海林美琴\.claude\permission-review-log.md` に追記

  - 関連ファイル・ブランチ名

## 11. 横断調整セッション運用（2026-04-23 追加）

### 役割分担

Gardenシリーズは複数のClaudeセッションが並行稼働するため、以下の役割分担で運用する：

| セッション | 役割 |
|---|---|
| **a-main（横断調整セッション）** | 全モジュール横断の指示・ルール改訂・セッション間の整合性調整・ユーザーからの全体方針伝達 |
| **a-soil / a-root / a-tree / a-leaf / a-bud / a-bloom / a-seed / a-forest / a-rill** | 担当モジュールの実装作業 |
| **b-main** | 東海林Bアカウントでの作業セッション（A使用枠到達時の引き継ぎ先） |

### 横断指示の受け取り方

他セッションから **「【横断セッション(a-main)からの共有】」** で始まるメッセージを受け取ったら：

1. **指示内容を最優先で確認**する（現在の作業の区切りで一度手を止める）
2. 影響を受けるモジュール・アクション要否を判断
3. 必要なアクションを実施
4. **疑問点があれば横断セッション経由でユーザーに確認**を依頼する（他モジュール実装セッション同士で勝手に合意しない）

### 恒久ルール vs 一時指示

| 種別 | 伝達経路 | 反映タイミング |
|---|---|---|
| **恒久ルール** | CLAUDE.md（親/子）・MEMORY.md に直接記載 | 次回各セッション起動時に自動反映 |
| **一時指示** | 「【横断セッション(a-main)からの共有】」テンプレート形式でユーザーが各セッションに手動コピペ | その場で即時適用（セッション内のみ） |

### 横断調整セッションからの発信判断基準

横断調整セッション側（a-main）は以下の基準で発信方式を決める：

- モジュール仕様の正式変更・新ルール追加・恒久的な合意事項 → **CLAUDE.md / MEMORY.md を直接編集**
- 当日の作業優先順位・一時注意・他セッションへの確認依頼 → **テンプレート形式でユーザーに渡す**

## 12. 作業工程の実績蓄積ルール（2026-04-23 追加）

### 目的

横断調整セッション（a-main）が、ユーザーから突発的に「〇〇モジュールの進捗は？」「全体の遅延はどれくらい？」と聞かれた時に**即答できる**ようにするため、各セッションは作業実績を共通ファイルに蓄積する。

### 蓄積先ファイル

`docs/effort-tracking.md`（リポジトリ直下の `docs/` 配下）
※ 既存の MEMORY.md `feedback_effort_tracking` を踏襲

### 記録タイミング

以下の区切りで随時追記する：
- 新機能・新フェーズ着手時（予定時間を記録）
- 機能・フェーズ完了時（実績時間・差分を記録）
- 方針変更・見積もり修正時
- 日次作業終了時（未完了タスクの中間記録）

### 記録フォーマット

`docs/effort-tracking.md` に以下の表形式で追記：

| 日付 | モジュール | タスク | 予定時間 | 実績時間 | 差分(±h) | 理由 | 担当セッション |
|---|---|---|---|---|---|---|---|
| 2026-04-23 | Garden-Tree | Phase A 認証画面実装 | 4h | 6h | +2h | RLS設計で詰まった | a-tree |
| 2026-04-23 | Garden-Forest | v9 レビュー修正 | 2h | 1.5h | -0.5h | 軽微修正のみで済んだ | a-forest |

### 運用ルール

- 作業区切りで**各セッションが追記 → `git commit & push`**
- A/B セッション共通（東海林A・B どちらでも同じファイルに追記）
- 追記したら軽量な commit メッセージでOK（例：`docs: effort-tracking 更新 (tree)`）
- セッション名（a-tree / a-forest / b-main 等）は末尾列に必ず明記

### 突発報告への対応（a-main の責務）

ユーザーから「〇〇の進捗は？」「全体の遅延は？」等の確認があった際、a-main は以下の順で即答する：

1. `git pull` で `docs/effort-tracking.md` を最新化
2. 該当モジュール・期間の行を抽出
3. **予定 / 実績 / 差分（±h） / 遅延理由** を表形式で報告
4. 必要なら全体サマリ（合計予定 vs 合計実績）も併記

### 初回導入時の対応

このルールが新規追加された時点で、各セッションは以下を実施：
- 現在進行中のタスクがあれば、**過去分を遡って記録**（把握できる範囲でOK）
- 未来分（直近予定タスク）を**予定時間付きで先行記入**

## 13. 自律実行モード（2026-04-24 改訂 v2 / 初版: 2026-04-23）

### 🔒 専用セッション方式（核心）

自律実行モードは **a-auto セッション専用** で発動する。a-auto 以外のセッション（a-main / a-forest / a-tree 等）では発動しない。

### 目的

ユーザー（東海林さん）が**確認作業できない時間帯**（就寝中・会議中・外出中・集中別作業中）に、5時間使用枠を有効活用して作業を進め、復帰時にレポートとして結果を受け取る運用モード。

### セッション役割分担

| セッション | 自律実行モード | 役割 |
|---|---|---|
| a-main（横断調整） | 発動しない | ユーザー対話・朝の判断・**周知配布支援** |
| **a-auto（自律実行専用）** | **★ここのみで発動** | 夜間・会議中の自律作業全般 |
| a-soil / a-root / a-tree / a-leaf / a-bud / a-bloom / a-seed / a-forest / a-rill | 発動しない | インタラクティブ作業専用（**周知受信役も兼ねる**） |
| b-main | 発動しない | 東海林Bバックアップ |

### 想定シーン

| シーン | 想定時間 | 補足 |
|---|---|---|
| 就寝前 | 5〜8時間 | 5時間使用枠をフル活用。朝レビュー |
| 会議中 | 30分〜2時間 | 会議前に発動、終了後に確認 |
| 外出中 | 1〜4時間 | 移動・接客などで確認できない時間 |
| 集中別作業中 | 任意 | 別タスクに集中している間、並行処理 |

### 発動方法（ユーザー操作）

**必ず a-auto セッション**に対し、次のいずれかを入力：

- 標準形：「`自律実行モード発動してください`」
- シーン明示形：「`自律実行モード発動してください。就寝前`」 / 「`会議中（約〇時間）`」 / 「`外出中（約〇時間）`」
- 対象モジュール指定形：「`自律実行モード発動してください。対象: a-tree, a-forest`」
- 優先指定形：「`自律実行モード発動してください。優先度: 〇〇`」
- タスク明示形：「`自律実行モード発動してください。タスク: 1)〇〇 2)△△`」

a-auto 以外のセッションに同じ発動指示が来た場合、「自律実行モードは a-auto 専用です。a-auto 側で発動してください」と応答して停止。

### 🔓 ユーザー不在時の許可モード設定（2026-04-24 追記）

就寝中・会議中・外出中など**ユーザーが画面を見られない時間帯**に発動する場合、以下を**発動前に必ず**実施：

1. a-auto セッションの入力欄で **Shift+Tab** を押す
2. 画面下に「⏵⏵ accept edits on」や「accept all edits」等の表示が出るまでサイクル
3. このモードでは**すべての許可確認が自動承認**される
4. その状態で自律実行モード発動プロンプトを投入

### 安全性の担保

Shift+Tab モード使用時でも、以下は依然として保護される：

- `.claude/settings.json` の **deny ルール**は最優先で適用
  - `rm` / `rmdir` / `Remove-Item` / `del`（削除系）
  - `git reset --hard` / `git push --force` / `git push -f` / `git clean -f`（Git破壊系）
  - `git branch -D`（ブランチ強制削除）
  - `git push origin main` / `git push origin develop`（保護ブランチへの直接push）
- Shift+Tab は**許可確認をスキップするだけ**で、deny を無効化しない
- つまり、対策A（settings.json 拡張）＋対策B（Shift+Tab）の**二重保険**で安全運用可能

### セッション単位の有効範囲

- Shift+Tab モードは**現セッション限り**
- セッション再起動後は通常モード（プロンプトあり）に戻る
- ユーザーが対面で作業する際は通常モード（発動時点でShift+Tabを押さない or 再起動）

### a-auto の動作フロー

1. 発動を受けて、対象モジュールまたはコンテキストを把握
2. 対象モジュールに該当するブランチに `git checkout`（または a-auto 作業専用ブランチ `feature/xxx-yyy-auto` を新規作成）
3. 他モジュールセッションが同ブランチで作業中でないか確認（直近コミット時刻・handoff 記述から推定）
4. 作業を実行（§13の制約内）
5. commit + push
6. 次の対象モジュールへ移動、同様の処理
7. 全タスク完了または停止条件で終了
8. **復帰時レポート + 周知メッセージ + 全体サマリ**を生成（Phase 1 で詳細規定）

### 他モジュールセッションとの協調

a-auto が作業するとき、**対象モジュールセッションは待機**が原則（ブランチ競合防止）。

- 発動時：ユーザーが対象モジュールセッションに「a-auto で自律実行中、このセッションは待機」と伝える
- 完了後：a-auto → a-main 経由で Phase 2 の短文配布 → Phase 3 の5ステップで復旧

補助策：**a-auto は可能な限り自作業専用ブランチ（`feature/xxx-yyy-auto`）を作成**することで、モジュール側の既存ブランチと衝突しない運用を推奨。

### 自動コンテキスト把握

a-auto は発動時、対象モジュールごとに以下を読み込む：

1. 対象モジュールに対応するブランチの直近コミット（`git log --oneline -20`）
2. `docs/handoff-*.md` / `docs/wip-*.md` の最新ファイル
3. `docs/effort-tracking.md` 内の対象モジュール担当分
4. 自セッション名＝`a-auto`（複数モジュールまたぎ前提）

### タスクリスト自動生成の優先順

1. ユーザーが発動時に明示した優先タスク・対象（最優先）
2. 対象モジュールの `docs/wip-*.md` / `docs/handoff-*.md` の「次にやるべきこと」
3. `docs/effort-tracking.md` で遅延（差分 +h）のあるタスク
4. 直近コミットから推測される論理的次手
5. 上記で具体化できない場合 → **停止**

### 判断ルール（迷ったらこうする）

- 設計方針の判断が必要になったら → **即時停止**、`docs/pause-YYYYMMDDHHMM-a-auto.md` 作成
- 既存コードへの変更：動作を変えない範囲のみOK、動作変更は設計判断扱い → 停止
- テストが通らない → 失敗理由を記録して次のタスクへ
- エラー解決に3回以上失敗 → 該当タスクを飛ばす
- ユーザーへの確認事項が発生 → 即停止
- 仕様解釈に幅あり → 既存実装・既存ドキュメントに最も近い選択

### 禁止事項

- ファイル・レコード削除（`.claude/settings.json` の deny でシステム的にも禁止済み）
- `main` / `develop` ブランチへの直接 push
- Supabase 本番データへの書き込み（書き込みが必要なタスクは停止）
- kintone のレコード追加・編集・削除
- 新規 npm パッケージのインストール
- 勝手な仕様変更・設計変更
- ユーザー事前承認なしの外部API課金呼び出し
- 他モジュールセッションが作業中のブランチへの同時push
- CLAUDE.md / MEMORY.md の編集（恒久ルール変更は判断事項）

### コミット方針

- 対象モジュールの作業ブランチ（`feature/xxx-yyy-auto` 等）で commit
- メッセージに **[a-auto]** タグを含める（例：`test(bud): [a-auto] 振込CRUDテスト追加`）
- push は作業の区切りで（最低でも復帰前に1回）
- 作業実績は対象モジュールフォルダの `docs/effort-tracking.md` に追記（§12準拠）

### 停止条件

1. 初期タスクリスト全件完了
2. 5時間使用枠が**90%に到達**
3. 設計判断・ユーザー確認事項が発生
4. 同じエラーに3回以上詰まる
5. タスク生成時にコンテキストが不足して判断できない
6. シーン別の想定時間超過（会議中・外出中は明示された時間を超えたら安全停止）
7. 就寝前シーン限定：3時間経過で安全停止

**対象モジュールごと**に個別レポートを出力：
- `<対象モジュールフォルダ>/docs/autonomous-report-YYYYMMDDHHMM-a-auto-<対象モジュール名>.md`

加えて、a-auto フォルダに**集約サマリ**を出力：
- `C:\garden\a-auto\docs\broadcast-YYYYMMDDHHMM\summary.md`


## 14. 許可リスト棚卸し運用（2026-04-24 追加）

### 目的
「一度だけ許可」を頻繁に押すコマンドを定期的に集計し、安全なものを恒久 allow リストに昇格させることで、ユーザーの✋確認回数を継続的に減らす。

### 🤖 自動発動ルール（a-main の責務）

a-main セッションが**各日最初の対話を受けた時**、以下を自動実行：

1. `G:\マイドライブ\..._東海林美琴\.claude\last-permission-review.txt` を読み込み
2. 最終棚卸し日から**7日以上経過**していたら → 棚卸しフロー発動
3. 経過していなければ → silent skip（ユーザーに何も表示しない）
4. 発動時は、**ユーザーが自発的に依頼していなくても自動で集計・提案**する

### 手動発動（on-demand）

ユーザーが a-main で「**許可リスト棚卸し**」と入力した場合、7日経過に関係なく即時発動。

### 棚卸しフロー

1. `~/.claude/projects/*.jsonl` の直近50セッションをスキャン
2. Bash / MCP ツール呼び出しの使用頻度を集計
3. 既存 allow リストに含まれず、auto-allow でもない、**頻出（3回以上）**コマンドを抽出
4. 安全性判定：
   - 非破壊・読み取り系 → **昇格候補**
   - 破壊系（rm / git reset --hard 等）→ **deny 強化候補**
   - 判断迷い → **保留リスト**
5. 結果を表形式でユーザーに提示（なければ「今週は追加候補なし」の1行報告）
6. ユーザー承認後、**14箇所の settings.json に一括反映**
7. 結果を `01_東海林美琴\.claude\permission-review-log.md` に追記
8. タイムスタンプ更新（`last-permission-review.txt`）

### 提示フォーマット

```
【週次許可リスト棚卸し】YYYY-MM-DD
直近1週間のスキャン結果：

■ 昇格候補（非破壊・3回以上）
| # | パターン | 回数 | 判定 |
|---|---|---|---|
| 1 | Bash(xxx *) | 5 | 非破壊・頻出 |
| 2 | ... | ... | ... |

■ deny 強化候補（破壊系）
| # | パターン | 回数 | 備考 |
| ... |

■ 保留（判断迷い）
| # | パターン | 回数 | 迷いポイント |
| ... |

承認/却下/保留の指示をお願いします。
```

### 関連ファイル

- **タイムスタンプ**: `01_東海林美琴\.claude\last-permission-review.txt`
- **履歴ログ**: `01_東海林美琴\.claude\permission-review-log.md`

### 運用ルール

- 自動実行は a-main のみ（他モジュールセッションでは発動しない）
- 結果は Python 一括更新スクリプトで全14箇所（01_東海林美琴 + Garden 13箇所）に反映
- deny 強化の提案は昇格とは独立した判断事項として別表示
- 変更適用後、各セッションは再起動 or 手動リロードで settings.json 変更を反映

## 16. リリース前バグ確認ルール（2026-04-24 追加）

### 目的
各モジュールの現場リリース前に、**autoモードで網羅的バグ探索**を行い、本番投入の信頼性を確保する。

### 7種の必須テスト（auto発動で実施）

1. **機能網羅テスト**：全ボタン・全リンク・全入力パターン（Playwright等）
2. **エッジケーステスト**：空入力・極大入力・特殊文字・絵文字・マルチバイト
3. **権限テスト**：7段階ロール（toss/closer/cs/staff/manager/admin/super_admin）別アクセス範囲確認
4. **データ境界テスト**：NULL / 負数 / 文字列最大長 / 日付境界
5. **パフォーマンス計測**：ページロード時間 / API応答時間
6. **コンソールエラー監視**：開発者ツール Console のエラー・警告
7. **アクセシビリティ確認**：axe-core / Lighthouse スコア

### 成果物

`docs/pre-release-test-YYYYMMDD-<module>.md` に：
- テスト実施サマリ（合格/警告/不合格の件数）
- 不合格・警告項目の詳細（再現手順・影響範囲）
- 修正優先度マトリクス（🔴緊急 / 🟡推奨 / 🟢後回し）

### モジュール別の厳格度

- **Tree（コールセンターの要）**: 🔴 **最厳格**。全7種テスト完走 + 1週間ベータ運用必須
- **Bud（経理、金銭関連）**: 🔴 厳格。特に金額・振込系のエッジケース必須
- **Root（認証・マスタ）**: 🔴 厳格。権限テスト・セキュリティ特に重視
- **Leaf（商材データ）**: 🟡 通常。データ整合性重視
- **その他**: 🟡 通常

## 17. 現場フィードバック運用ルール（2026-04-24 追加）

### 目的
段階的な現場投入と、フィードバックのナレッジ化で**同じミスを繰り返さない**仕組み。

### 3段階のテスト運用

| 段階 | 対象 | 期間 | 内容 |
|---|---|---|---|
| **α版** | 東海林さん1人 | 即時 | 機能完成時の即時確認、修正即反映 |
| **β版** | 3-5人（信頼できるスタッフ） | 1-2週間 | 実業務で試用、フィードバック収集、修正 |
| **リリース版** | 全社展開 | - | 運用開始、段階的に機能拡張 |

### Tree 特例（コールセンター現場投入）

Tree は FileMaker 稼働中の主力業務のため、**超慎重な展開**：

1. **α版**: 東海林さん1人で徹底確認（§16 の7種テスト完走後）
2. **1人現場テスト**: コールセンタースタッフ1名で1週間（新旧並行、問題なければ継続）
3. **2-3人テスト**: 1週間
4. **半数テスト**: 1-2週間
5. **全員投入**: FileMaker 切替

### フィードバック収集

- **収集先**: Chatwork / Google Form / 口頭 → 集約
- **記録**: `docs/field-feedback-YYYYMMDD-<module>.md`
- **分類**: 🐛バグ / 💡UX改善要望 / ✨新機能要望 / ⚠️重大（業務停止級）

### 予防策ナレッジ化

- `docs/known-pitfalls.md` に蓄積（全モジュール横断）
- 新モジュール構築時、このファイルを**必読**にする
- autoモードのバグ確認プロンプトにも、このナレッジを反映

### フィードバック反映サイクル

```
α版 → β版 → リリース
  ↓      ↓       ↓
バグ検出 → バグ修正 → 予防策を known-pitfalls に追記
        ↓
次モジュール構築時に参照
```
