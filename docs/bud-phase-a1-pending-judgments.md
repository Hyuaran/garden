## 🔴 即時合意 判断要約（7 件）

### A-03 判1: 状態遷移履歴の記録先

| 項目 | 内容 |
|---|---|
| 論点 | 振込の状態遷移履歴を Bud 独自テーブルに記録するか、既存の全社横断監査ログに統一するか |
| 候補案 | **A) 両方に記録（二重化）** / B) Bud 独自 `bud_transfer_status_history` のみ / C) `root_audit_log` のみに統一 |
| 推奨案と理由 | **A（両方記録）** — Bud 内参照（詳細画面の履歴タブ）は `bud_transfer_status_history` が高速、横断監査（全モジュール SOX 対応）は `root_audit_log` を使う二層構成で役割分離 |

### A-03 判3: super_admin 自起票時の reason 自動挿入

| 項目 | 内容 |
|---|---|
| 論点 | super_admin が自分で起票し即承認するとき、履歴テーブルの `reason` カラムに何を記録するか |
| 候補案 | A) NULL（空のまま） / **B) 「自起票」固定文字列** / C) 都度手入力必須 |
| 推奨案と理由 | **B（「自起票」固定）** — 将来の監査で通常承認と即承認を区別可能、かつ super_admin の入力負担ゼロを両立 |

### A-04 判1: 新規取引先の追加 UI 位置

| 項目 | 内容 |
|---|---|
| 論点 | 振込フォーム入力中に新規取引先を追加したい場合、どこで追加させるか（フォーム離脱リスクとマスタ責務の分離のトレードオフ） |
| 候補案 | **A) Bud 内モーダルで追加** / B) Root マスタ画面へ遷移 / C) Root 側で事前追加必須（Bud からは追加不可） |
| 推奨案と理由 | **A（Bud 内モーダル）** — フォーム離脱を防ぎ起票効率を確保。Root 側の `root_vendors` への INSERT RLS を admin+ に限定して責務を担保 |

### A-04 判2: 当日振込の扱い

| 項目 | 内容 |
|---|---|
| 論点 | 振込予定日に「当日」を指定できるようにするか |
| 候補案 | **A) 翌営業日以降のみ許可** / B) 当日も許可 / C) admin のみ当日選択可 |
| 推奨案と理由 | **A（翌営業日以降のみ）** — ネットバンキングの当日振込は手数料・締め時間の個別判断が発生するため、当日分は Garden 外（admin の個別対応）で処理 |

### A-05 判1: 本人承認禁止の厳密化方針

| 項目 | 内容 |
|---|---|
| 論点 | 自分が起票した振込を自分で承認することを UI/DB でどこまで制限するか（super_admin 自起票スキップとの整合性） |
| 候補案 | **A) 警告のみ表示、操作は可能** / B) UI 側で承認ボタン非活性 / C) DB RLS で完全禁止 |
| 推奨案と理由 | **A（警告のみ）** — super_admin の自起票スキップを許容する運用と整合。Phase A は警告で開始、Phase B で運用実績を見て厳密化（B or C）を再判断 |

### A-06 判1: 銀行明細 CSV の対応範囲

| 項目 | 内容 |
|---|---|
| 論点 | 銀行明細取込で Phase A 時点で対応する CSV 形式の範囲 |
| 候補案 | **A) 楽天・みずほ 2 種のみ** / B) 4 種（楽天/みずほ/PayPay/京都銀行） / C) 全銀協統一形式のみ |
| 推奨案と理由 | **A（楽天・みずほ 2 種）** — 実運用で明細件数が多い 2 行から着手、ROI を早期化。PayPay・京都銀行は需要を見て順次追加（パーサ追加のみで拡張可） |

### A-08 判1: CC 明細 対応カード種類

| 項目 | 内容 |
|---|---|
| 論点 | Phase A 時点で取込対応する CC 種類 |
| 候補案 | **A) 楽天ビジネス 1 種のみ** / B) 楽天 + JCB + Amex 3 種 / C) 全 CC（オリコ・NTTBiz・三井住友 2 種含む） |
| 推奨案と理由 | **A（楽天ビジネス 1 種）** — 最多利用カードでインボイス取得運用を先に確立、JCB・Amex・オリコ・NTTBiz・三井住友は形式確認後に順次（Phase 2b で全件展開予定） |
