## 改訂版（v5.2、5/11 提案）

```
## 5. dispatch 形式 v5.2

memory `feedback_dispatch_header_format` v5.2 + `docs/dispatch-template.md` 準拠。

### 5-1. 投下情報の冒頭明示（v5 継承）

- 投下先 / ファイル markdown link / 添付有無 / 緊急度 を**先頭に明示**
- コピペ本体は md ファイル経由（チャット内に出さない）
- 複数 dispatch 同時発行時はリスト化（# 1, # 2, # 3）
- 末尾に「投下完了したら『NNN投下済』で OK」依頼

### 5-2. 投下用短文 ~~~ ラップ内の禁止事項（v5.2、最重要）

投下用短文 ~~~ ラップ内に **同記号繰返しブロック（~~~ / バッククォート 3 個）を入れることを禁止**（コピペ分断防止）。

| 違反パターン | 規定 |
|---|---|
| 違反 7（v5.1 既往）: ~~~ 内にコードブロック（バッククォート 3 個） | 禁止 |
| 違反 7-b（v5.2 拡張）: ~~~ 内に同記号繰返しブロック（~~~ 自身） | 禁止（ネスト分断防止）|
| 違反 9（v5.2 新規）: 冒頭 3 行（番号 + 元→宛先 + 発信日時）を ~~~ 外配置 | 禁止、冒頭 3 行は ~~~ 内配置必須 |

### 5-3. 雛形ベストプラクティス（v5.2 新規、提案 2 由来）

dispatch 内に報告フォーマット雛形（audit-001- No. NN / analysis-001- No. NN 等）を提示する場合、~~~ ネスト + コードブロックは使わず以下方式で記述する:

| 方法 | 内容 |
|---|---|
| ### サブヘッダーで構造を示す | 「### 件名」「### 内容」「### self-check」等 |
| 例示部分は通常 markdown 記述 | 表 / 箇条書き / 引用ブロック（`>`）|
| HTML / コード例を示す必要がある場合 | 外ラップ ~~~ を撤廃してから提示、または別 md ファイル化 |

### 5-4. 既存 dispatch の修正方針（v5.1 継承）

| 状況 | 動作 |
|---|---|
| 投下未済 dispatch | 投下前に v5.2 修正 |
| 投下済 dispatch | 過去分は触らない、本ルールは新規起草分から適用 |
```

## 改訂理由

- 違反 7（v5.1 既往）の規定は memory のみだったが、a-main-018 期に多重発生（audit-001- No. 2 critique）= governance レベル明文化が必要
- 違反 9（冒頭 3 行 ~~~ 外配置）が a-main-020 初期 dispatch でも未明文化のまま運用 → v5.2 で明記
- 雛形ベストプラクティスは a-audit / a-analysis の report 起草で頻出 → 標準化必要
