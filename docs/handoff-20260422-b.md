### 💡 重要な設計判断（既決）
- **振込 ID 形式**: Kintone 既存運用継承（通常 `FK-YYYYMMDD-NNNNNN` / キャッシュバック `CB-YYYYMMDD-G-NNN`）。Phase 0 schema の `FRK-` は廃止
- **6 段階ステータス**: 下書き → 確認済み → 承認待ち → 承認済み → CSV出力済み → 振込完了（+ 差戻し）
- **super_admin スキップ**: 東海林さんが自分で起票した振込は「即承認」ボタンで `下書き → 承認済み` に 1 クリック遷移。**このとき `confirmed_by / confirmed_at` は NULL のまま残す**（「二重チェックをスキップした」ことを明示・I-2 B 案採用）
- **社内立替対応**: `request_company_id`（依頼会社）と `execute_company_id`（実行会社）を分離。Kintone 実データに「リンクサポート依頼・ヒュアラン実行」のような立替事例があるため
- **重複検出**: PostgreSQL GENERATED 列（duplicate_key）+ TypeScript 側同ロジック。UI で入力中にリアルタイム警告表示
- **MF クラウド給与連携** は Phase 6 実装時（CSV 形式取込済み、72 列）
