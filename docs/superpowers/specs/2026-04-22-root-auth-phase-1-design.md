# Garden-Root 認証・権限管理 Phase 1 設計書

作成日: 2026-04-22
作成者: 東海林B セッション（Claude Code）
バージョン: 1.0
対象モジュール: Garden-Root

---

## 1. 目的

Garden-Root（経営マスタ管理）に **ログインと権限チェックの仕組み** を導入する。
現状は誰でも 7 マスタ画面を閲覧・編集できる状態のため、機密性の高いマスタデータ（従業員の給与振込口座など）を適切に保護する。

## 2. スコープ

### このフェーズに含む
- Root 専用ログイン画面（社員番号 + パスワード）
- 権限による画面閲覧制御（責任者以上のみ閲覧可）
- 権限による編集制御（管理者以上のみ編集可）
- 監査ログ記録
- セッション自動ログアウト（2 時間無操作）
- ログアウト後の元画面復帰

### このフェーズに含まない（後続フェーズで対応）
- 他アプリから Root データへの参照ルール整備（Phase 2）
- KING OF TIME 連携（Phase 3）
- Root ↔ Forest 法人連携表（Phase 4）
- MF クラウド給与連携（Phase 5）
- 監査ログの閲覧画面
- パスワード N 回失敗でロック
- IP 制限 / 2 段階認証
- 自動テスト（Jest / Playwright）

## 3. 画面と操作の流れ

### 初回アクセス
1. ブラウザで `/root` を開く
2. 未ログイン → 自動で `/root/login` へリダイレクト
3. ログイン画面で社員番号とパスワードを入力
4. ログイン成功 → 権限に応じて分岐

### 権限による分岐

| ロール | ログイン後の挙動 |
|---|---|
| `admin` / `super_admin` | Root トップへ遷移。閲覧・編集すべて可能 |
| `manager`（責任者） | Root トップへ遷移。閲覧のみ。編集ボタンはグレーアウト |
| `toss` / `closer` / `cs` / `staff` | 「アクセス権限がありません」画面。戻るボタンで `/` へ |

### 通常の作業中
- 画面右上に `👤 氏名（ロール表示） [ログアウト]` を常設表示
- 2 時間タイマーが稼働
- 残り 10 分で警告モーダル表示
- 2 時間経過で自動ログアウト

### 再ログイン
- ログアウトした画面の URL を記憶
- 再ログイン成功時、元の URL に戻る（例: `/root/employees` で切れたら `/root/employees` に戻る）

## 4. 権限マトリックス

### Garden 全体の 7 段階ロール
`toss` → `closer` → `cs` → `staff` → `manager` → `admin` → `super_admin`

### Garden-Root の権限

| ロール | Root ログイン | 閲覧 | 編集 | 新規登録 | 無効化 |
|:---:|:---:|:---:|:---:|:---:|:---:|
| toss / closer / cs / staff | ❌ | ❌ | ❌ | ❌ | ❌ |
| manager（責任者） | ✅ | ✅ | 🔒 | 🔒 | 🔒 |
| admin（管理者） | ✅ | ✅ | ✅ | ✅ | ✅ |
| super_admin（全権管理者） | ✅ | ✅ | ✅ | ✅ | ✅ |

### 拒否時の UX

- **ログイン拒否（staff 以下）**: 「アクセス権限がありません」画面 → 自動ログアウト → `/root/login` へ
- **編集ボタンの拒否（manager）**: ボタンはグレーアウト。tooltip「編集権限がありません（管理者以上）」
- **URL 直打ちの編集**: 画面側で拒否表示。保存 API も権限チェックで拒否

### 注記
- `garden_role` 自体の変更は `super_admin` のみ可（Phase 2 以降で UI 追加）
- レコードの物理削除は禁止。「無効化」扱い（`is_active = false`）

## 5. 監査ログ

### 記録先
`root_audit_log` テーブル（Tree Phase A で既に作成済み・全モジュール共通）

### 記録する 6 アクション

| # | アクション | 契機 |
|---|---|---|
| 1 | `login_success` | ログイン成功 |
| 2 | `login_failed` | 誤パスワード・存在しない社員番号 |
| 3 | `login_denied` | 権限不足で拒否（staff 以下） |
| 4 | `logout` | 手動ログアウトまたはタイマー切れ |
| 5 | `master_update` | 7 マスタの CRUD（新規・編集・無効化） |
| 6 | `permission_denied` | 責任者が書込 API を叩いた |

### 記録項目（全アクション共通）

- 操作日時（ミリ秒精度）
- 操作者の `auth.users.id`
- 操作者の社員番号
- アクション名（上記 6 種類）
- 対象テーブル（例: `root_employees`、認証系は空欄）
- 対象レコード ID（例: `EMP-0001`、認証系は空欄）
- 追加情報 JSON（変更前後の値・失敗理由など）
- IP アドレス
- User-Agent（ブラウザ情報）

### ポリシー
- `login_failed` は社員番号が存在しない場合も記録（不正アクセス検知用）
- マスタ編集は変更前後の値を全フィールド記録（保管容量は Phase 2 で再検討）
- 監査ログ書込失敗は main 処理を止めない（ブラウザコンソールにエラー出力のみ）

## 6. セッション管理

### タイムアウト仕様

- **本番設定**: 2 時間（最終操作から）
- **警告**: 残り 10 分で警告モーダル表示
  - ラベル: 「作業を続ける」「ログアウト」
  - モーダル表示中も操作があればタイマーリセット + モーダル閉じる
- **自動ログアウト**: 2 時間経過時

### 「最終操作」の定義
- キーボード入力
- マウスクリック
- ページ遷移

### 開発モードでの短縮タイマー

**⚠️ 本番リリース前に必ず元に戻すこと**

```typescript
// src/app/root/_lib/session-timer.ts（実装予定）
const SESSION_TIMEOUT_MS =
  process.env.NODE_ENV === "development"
    ? 30 * 1000          // 開発時: 30 秒
    : 2 * 60 * 60 * 1000;  // 本番: 2 時間

const WARNING_OFFSET_MS =
  process.env.NODE_ENV === "development"
    ? 10 * 1000          // 開発時: 残り 10 秒で警告
    : 10 * 60 * 1000;    // 本番: 残り 10 分で警告
```

- 開発時（`npm run dev`）は 30 秒タイムアウト・残り 10 秒で警告
- 本番デプロイ時（Vercel）は 2 時間・残り 10 分
- 開発モード時は警告モーダルに「⚠️ 開発モード中」バッジを表示

### 複数端末・複数タブ
- 同じアカウントで別端末ログインは Supabase Auth の標準動作に任せる（後勝ち）
- 複数タブでログアウトした場合、他タブもログアウト画面に数秒以内で遷移

## 7. エラー処理

| # | 状況 | 表示 | アクション |
|---|---|---|---|
| 1 | 通信エラー中のログイン | 「通信エラーが発生しました。ネットワークをご確認ください」 | [再試行] |
| 2 | 誤パスワード | 「社員番号またはパスワードが正しくありません」 | 再入力（ロックなし） |
| 3 | 権限不足 | 「Garden-Root へのアクセス権限がありません。責任者以上の方にお問い合わせください」 | [戻る] → `/` |
| 4 | セッション切れ | 「操作時間を超過したためログアウトしました」 | [ログイン画面へ] |
| 5 | ロール変更検知 | 「権限が変更されたためログアウトします」 | [ログイン画面へ] |
| 6 | マスタ保存失敗 | 画面上部の赤帯「保存に失敗しました：<理由>」 | 入力内容保持・[再試行] |
| 7 | 他タブの自動ログアウト反映 | 「別タブでログアウトされました」 | [ログイン画面へ] |

### 監査ログ書込失敗
- main 処理は成功させる（ログイン・マスタ保存など）
- コンソールにエラーログ
- 連続失敗時の自動アラートは Phase 2 以降で検討
