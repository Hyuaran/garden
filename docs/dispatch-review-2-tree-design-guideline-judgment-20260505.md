| 2 | 円リング進捗バー 不採用 | 🟡 修正推奨 | toss/closer 個人 UI で不採用は妥当、ただし **manager+ KPI ダッシュボード**では転用検討余地あり（後述） |

【Q1 Today's Activity 右サイドバーの扱い → B 案推奨】

| 案 | 推奨 | 理由 |
|---|---|---|
| A 完全省略 | ❌ | Garden ファミリー統一感が崩れ、後道代表に「Tree は別アプリ」認知リスク |
| **B 最小表示** | ✅ | 業務集中 + Bloom 統一感の両立、トレードオフの最適点 |
| C デフォルト展開 + 折畳可 | ❌ | 折畳判断を毎回オペレーターに委ねるのは認知負荷、デフォルトが業務阻害 |

**B 案の最小表示 設計指針**:

含めるもの（架電業務に直結）:
1. 自分の状態インジケータ（待機中 / 架電中 / 休憩中 / トス送信中）
2. 通知センター（**重要通知のみ**：トス受領完了 / 上司からの個別指示 / 緊急アラート）
3. 折畳トグルは保持（Bloom ヘッダー要素統一性のため、草案 §3-4 と整合）

含めないもの（業務阻害可能性あり）:
- ❌ closer 空き状況・closer 待ち時間（**memory `project_tree_toss_focus_principle.md` 違反**、toss 視点での closer 可視化は禁止）
- ❌ 他人の活動ログ（視覚ノイズ）
- ❌ スケジュール / 予定（架電中は不要）
- ❌ システム情報（管理者 UI に分離）

→ 草案 §4-2 を「Bloom: デフォルト展開 / Tree: **B 案 最小表示（自分の状態 + 重要通知のみ、closer 状況非表示）**」に確定推奨。

【Q2 円リング進捗バー / 勤務形態バッジ → 個人 UI 不採用、manager+ で転用検討】

**結論: 草案の暫定不採用は妥当、ただし §4-1 に転用可能性を補足推奨**。

理由（memory `project_tree_d2_release_strategy.md` Track B 連動）:

- **Track A 架電系統 (toss/closer/cs/staff)**: 個人 UI には円リング・勤務バッジ不要、草案通り不採用
- **Track B 責任者の〆作業ツール (manager/admin)**: 「Garden すごい」感醸成のため**視覚的インパクト推奨** → 円リング (KPI 達成率) + 勤務バッジ (出社/在宅状況俯瞰) は積極的に転用可

具体的転用候補:
- 円リング: 個人 KPI ダッシュボード（架電数 / トスアップ率 / 成約率の達成率）、チーム月次目標達成率
- 勤務バッジ: 在宅対応オペレーターがいる場合の管理者向け稼働俯瞰 UI、cs ロールの勤務状況確認

→ 草案 §4-1 不採用テーブルに「**ただし Track B 責任者向け UI（manager+）では転用検討可**」の脚注追記を推奨。a-tree 資料化の権限マトリクス（toss/closer/cs/staff/manager/admin/super_admin × 画面）受領後、manager+ 専用画面が浮かんだ時点で個別判断。

【Q3 細部（フォント / padding / コントラスト / アニメーション）→ ハイブリッド方針推奨】

**結論: 一部は先決、一部は a-tree 資料化後の試作で視覚判断**。

| 項目 | 判定タイミング | 理由 |
|---|---|---|
| WCAG AA コントラスト | **先決**（規定固定）| 業務利用の最低ライン、実装段階で個別 PASS 確認は普通の作業 |
| 過度なアニメーション禁止 | **先決**（規定固定）| 草案 §2-2 厳守事項 #3 で確定済、追加判断不要 |
| フォントサイズ「大きめ」の具体値 | **試作後判断** | 14px? 16px? 18px? 実画面で電話番号 / トーク履歴を見ないと決められない |
