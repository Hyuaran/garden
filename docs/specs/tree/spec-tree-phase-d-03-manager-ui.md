# Tree Phase D-03: マネージャー / スーパーバイザー UI

- 対象: コールセンターマネージャー（manager / admin+）が使う監督・介入ダッシュボード
- 優先度: **🔴 高**（オペレーター業務の品質管理に必須）
- 見積: **0.8d**
- 担当セッション: a-tree
- 作成: 2026-04-25（a-auto / Batch 9 Tree Phase D #03）
- 前提:
  - D-01（テーブル 3 つ、RLS マネージャー権限）
  - D-02（オペレーター UI、session / call の INSERT 動作確定）
  - spec-cross-rls-audit（部署別 SELECT の `is_same_department`）
  - spec-cross-chatwork-notification（Batch 7、介入メッセージ送信経路）

---

## 1. 目的とスコープ

### 目的

コールセンターマネージャーが **リアルタイムで全オペレーター稼働を把握**、必要に応じて**介入（モニタリング・指示メッセージ送信）**できる UI を提供する。

### 含める

- リアルタイム稼働状況（在席 / 通話中 / 離席 / 休憩）
- オペレーター別パフォーマンス（コール数 / 成約率 / 稼働時間）
- 介入機能（モニタリング / 指示メッセージ送信）
- 録音聴取（§録音仕様 = 判断保留）
- 当日 KPI サマリ（部署別 / 個人別）
- 週次 / 月次レポートへの導線（D-05 と連携）
- アラート（低成約率・離脱多発・長時間離席）

### 含めない

- 月次 KPI 集計 VIEW / エクスポート（D-05）
- オペレーター側 UI（D-02）
- トスアップ追跡詳細（D-04、ここではサマリのみ）

#### 表示項目

| 列 | ソース | 更新頻度 |
|---|---|---|
| 氏名 | `root_employees.name` | 変わらない |
| 状態 | `v_tree_operator_today` + `tree_calling_sessions.mode` | 30s |
| コール数 | `tree_call_records` count (当日) | 30s |
| トス数 | `result_code = 'toss'` count | 30s |
| 成約率 | toss / connected | 30s |
| アラート | §6 アラート判定結果 | 30s |

#### 状態判定

```typescript
function getOperatorStatus(session, lastCalledAt): OperatorStatus {
  if (!session) return 'offline';
  if (session.mode === 'breeze' && lastCalledAt > now - 60s) return 'calling';
  if (lastCalledAt > now - 5min) return 'active';
  if (lastCalledAt > now - 15min) return 'idle';  // 離席
  return 'break';  // 休憩扱い
}
```

### 3.3 介入 UI

#### モニタリング

- 「モニタリング開始」押下で別タブ開き、オペレーターの画面を読み取り専用で表示
- 実装案 A: WebSocket で画面状態を push（実装コスト高）
- 実装案 B: 30 秒 interval で `v_tree_operator_today` を表示（実装コスト低、粒度粗い）
- **推定スタンス**: Phase D-1 は案 B、D-2 以降で WebSocket 検討

#### 指示メッセージ送信

## 4. リアルタイム更新

### 4.1 方式

- 30 秒間隔の polling（`v_tree_operator_today` に SELECT）
- WebSocket は使わない（Phase D-1 スコープ外）
- `visibilitychange` 検知でタブ非アクティブ時は polling 停止（帯域節約）

### 4.2 実装

```typescript
useEffect(() => {
  if (!document.hasFocus()) return;
  const id = setInterval(() => {
    supabase.from('v_tree_operator_today').select().then(setOperators);
  }, 30_000);
  return () => clearInterval(id);
}, [visible]);
```

### 4.3 負荷見積

- オペレーター 20 人 × マネージャー 3 人 × 2 polls/min = 120 req/min
- `v_tree_operator_today` は部分 index で < 100ms 応答（D-01 §8）
- 問題なし

## 6. アラート（自動検出）

### 6.1 検出対象

| アラート | 判定条件 | 対処 |
|---|---|---|
| 🔴 低成約率 | 当日コール 30+ & 成約率 < 5% | バッジ表示 + 指示送信ボタン強調 |
| 🔴 離脱多発 | 当日 NG クレーム 3+ | ダッシュボードに警告表示 |
| 🟡 長時間離席 | 最終コールから 30min+（通話外） | 状態バッジ 🔴 |
| 🟡 連続不通 | 10 件連続 `unreach` | リスト品質に問題？ バッジ表示 |
| 🟢 好調 | 成約率 > 20% & コール数 > 平均 1.5 倍 | 緑バッジ（動機付け）|

### 6.2 通知先

- ダッシュボード上でバッジ表示（即時）
- 重要度 🔴 は **Chatwork マネージャーチャンネルに自動通知**（§cross-chatwork 経由）
- 通知頻度制御: 同一オペレーターの同一アラートは 30 分に 1 回まで

## 8. Chatwork 連携

### 8.1 通知パターン

| トリガー | 宛先 | 本文 |
|---|---|---|
| 🔴 低成約率検出 | マネージャー DM | "◯◯さんの成約率が 5% 下回ります。Garden にて状況確認をお願いします。" |

| 週次サマリ | マネージャー DM | "先週のチーム KPI サマリを Garden Tree にて公開しました。" |

### 8.2 署名 URL 方針

**案 D 準拠**: Chatwork 本文には **Garden ログイン URL のみ**記載、署名 URL（機密）は流通させない。

### 8.3 レート制限

spec-cross-chatwork §6 準拠: 5 req/sec 制限、p-queue で逐次送信。

## 9. アクセシビリティ

### 9.1 マネージャーも非技術者

- オペレーター一覧テーブルは `aria-label` 付与、行は `role="button" tabIndex={0}`
- ソート変更時 `aria-live="polite"` で読み上げ
- アラートバッジ色だけに頼らず `aria-label="低成約率警告"` 併記

### 9.2 色覚対応

- 🟢🟡🔴 だけでなく形状（●▲■）併用、または文字併記
- 成約率グラフは色 + パターン塗り分け

## 11. テスト観点

詳細は D-06 §3。本 spec 固有観点：

- 30s polling 停止（非アクティブ時）
- 状態判定（active / idle / break / offline）の境界
- アラート 5 種の検出と通知送信
- RLS: マネージャーが他部署の session を見られないこと
- モニタリング: アクセス不可ユーザー（営業）からの拒否
- 指示送信: Chatwork 本文に署名 URL が含まれないこと（案 D 検証）

## 12. 判断保留事項

- **判1: 録音聴取の要否**
  - 録音ファイルが PBX 側にある場合、聴取方式は PBX ダッシュボード誘導 or Garden 内再生
  - **推定スタンス**: Phase D-1 は PBX リンク誘導のみ、Garden 内再生は D-2 検討
- **判2: モニタリング方式**
  - WebSocket vs polling
  - **推定スタンス**: Phase D-1 は 30s polling（WebSocket は D-2）
- **判3: アラート閾値**
  - 成約率 < 5% / コール数 30+ 等の具体数字
  - **推定スタンス**: 業務統計平均値の 50% を基準、初期 3 ヶ月で調整
- **判4: 指示メッセージのテンプレート化**
  - 「よく使う文面」を事前登録 / 毎回手動
  - **推定スタンス**: D-1 は手動、利用頻度見て D-2 でテンプレ化検討
- **判5: マネージャー間の介入権限**
  - 他部署マネージャーが別部署オペレーターに指示送信可能か
  - **推定スタンス**: 不可（RLS で自部署のみ）、全社横断は admin+ のみ
- **判6: アラート通知の ON/OFF 設定**
  - マネージャー個別にミュート可能にするか
  - **推定スタンス**: 可、`root_employees.alert_preferences` jsonb 列追加
- **判7: 複数セッション跨ぎの表示**
  - 1 日 2 セッション以上の場合、合算表示か個別表示か
  - **推定スタンス**: 合算を既定、個別は「詳細」リンクで開く
