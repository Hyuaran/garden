## 0. Track B 戦略の位置付け

### 0.1 目的（最重要、ブレずに）

**社員に「Garden すごいものができたぞ！」の第一印象を与えること**。

Track A（D-1+D-2 セットリリース）は §17 Tree 特例で慎重に段階展開する必要があるが、慎重さゆえに「すぐには Garden の効果が見えない」期間が発生する。この期間に **責任者の手作業 Excel を Garden が代行**することで、責任者経由で「Garden 入れたら楽になった」というポジティブな噂・体感を社内に蓄積する。

これにより、Track A の慎重展開期に **Garden への信頼感 = 抵抗感の低下**が達成され、Track A の 1 人 → 2-3 人 → 半数 → 全員の各段階の受入が円滑化する。

### 0.2 後道さんとの関係

- Track B は **営業側のツール**、後道さんは **経理・税理士パートナー**
- 後道さんは Track B を**見ない**（後道さん UX ゲートの対象外）
- 後道さん UX ゲートは別途 Track A の盆栽中心ビュー（cross-ui-04/06/01）で対応
- → **本 spec は後道さん FB 不要で進められる**（memory `feedback_ui_first_then_postcheck_with_godo.md` の架電画面例外と同方針）

### 0.3 工数管理

- Track A の +2.0d ソフトフォン実装（plan v3 の 6.5d → 8.5d）とは**別建て**
- Track B 単独で計上、effort-tracking には別行で記録
- 進行ペース: Track A 慎重展開期に並行進行、Track A の各段階終了時に Track B 機能を 1 つずつリリース

## 6. データソースとスキーマ依存

### 6.1 Tree D-01 schema との関係

本 spec の F1（実件数報告）と F3（サマリ画面）は、**Tree D-01 schema の `tree_call_records` 完成後に着手可能**。

選択肢：
- **案 A（推奨）**: Tree D-01 完成 → Track B 着手（schema 共用、追加マイグレーション最小）
- **案 B**: Track B 独自に簡略 schema を立てる（D-01 完成前でも先行可、ただし後で統合コストあり）
- **案 C**: 既存 FileMaker から Soil 経由で集計（Garden が架電 INSERT を握る前の暫定運用）

→ a-main 判断仰ぐ（判断保留 #3）

### 6.2 Bloom Phase A-1 との関係

Bloom が全社 KPI、Track B F3 が **コールセンター部門速報**として役割分担。
重複防止のため：
- Bloom は **月次集計 + 全社ビュー**を担当
- Track B F3 は **当日リアルタイム + 上位/下位 3 名**を担当
- Bloom が完成したら、Track B F3 の長期データは Bloom に移行（Track B は速報専用に絞る）
