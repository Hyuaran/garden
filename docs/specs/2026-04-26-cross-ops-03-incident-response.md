# Cross Ops #03: Garden 全体 インシデント対応設計

- 対象: Garden 全モジュールの障害発生時の対応プロセス
- 優先度: **🔴 高**（業務継続性、信頼性）
- 見積: **0.5d**
- 担当セッション: a-main + 各モジュール
- 作成: 2026-04-26（a-auto 003 / Batch 15 Cross Ops #03）
- 前提:
  - §17 現場フィードバック運用ルール
  - Cross Ops #01 monitoring-alerting / #02 backup-recovery
  - `docs/known-pitfalls.md`（既知の落とし穴ナレッジ）

---

## 1. 目的とスコープ

### 1.1 目的

Garden で**業務影響のあるインシデント**が発生した時、検知から終息・再発防止までを**標準化された手順**で進める。誰が / 何を / どの順で対応するかを明文化し、属人化を排除する。

### 1.2 含めるもの

- 重大度分類（Sev1 / Sev2 / Sev3）
- 重大度別の対応プレイブック（誰が / 何を / どの順で）
- インシデント宣言・終息のフロー
- ポストモーテム（事後分析）フォーマット
- 過去事例の蓄積（known-pitfalls 連動）

### 1.3 含めないもの

- 監視・検知の仕組み → #01
- バックアップ・リストアの技術詳細 → #02
- リリース時の不具合対応 → #04（リリース前検査）
- セキュリティインシデント単独の手順（一般インシデントとして扱う、機密度高い場合のみ別資料）

## 2. 重大度（Severity）分類

### 2.1 3 段階の定義

| Sev | 定義 | 例 | RTO 目標 |
|---|---|---|---|
| **Sev1** | 業務停止級 | 全社ログイン不可、振込実行が完全停止、給与配信失敗（締日当日）| 1h 以内 |
| **Sev2** | 一部機能停止 | 1 モジュール全停止、特定ロールのみ操作不可、Cron 連続失敗 | 4h 以内 |
| **Sev3** | 軽微 | UI 軽微なバグ、一部レコード表示崩れ、間欠的エラー | 1 営業日以内 |

### 2.2 重大度判定の早見表

```
障害発生 → 以下を順番に確認:

Q1. 全ユーザー / 全モジュールに影響？
  YES → Sev1
  NO  → Q2

Q2. 業務継続不可（金銭処理 / 給与 / 関電業務委託 等）?
  YES → Sev1
  NO  → Q3

Q3. 1 モジュール全体 or 主要機能停止？
  YES → Sev2
  NO  → Q4

Q4. 一時的回避策なし？
  YES → Sev2
  NO  → Sev3
```

### 2.3 Garden 固有の Sev1 例

- Supabase Auth サインイン全体不可
- Vercel デプロイ失敗 → サイトダウン
- DB 接続不能（Supabase 障害 or 接続枠超過）
- 振込承認画面の壊れ（経理締日当日）
- 給与明細 PDF 生成全失敗（給与配信締日）
- KoT 同期失敗が 24h 継続（勤怠データ消失リスク）

### 2.4 重大度の昇格・降格

- 当初 Sev3 と判定 → 影響拡大で Sev2 / Sev1 に昇格はあり得る
- 一時回避策で業務継続できれば Sev1 → Sev2 に降格可能
- 昇降格は **インシデント司令官（後述）が宣言**、Chatwork に明記

## 3. 役割定義

### 3.1 インシデント司令官（IC: Incident Commander）

- 1 件のインシデントに **1 名のみ**
- 既定: **東海林さん**
- 不在時のフェイルオーバー:
  - Sev1: 翌朝まで待たず代理 IC を立てる（暫定: super_admin の最初に応答した者）
  - Sev2/3: 翌朝まで待機可

#### IC の責務

- 重大度判定 / 昇降格宣言
- 対応者の招集 / 役割分担
- ステークホルダー通知（Chatwork）
- 終息宣言
- ポストモーテム主催

### 3.2 対応担当（Responder）

- 該当モジュール担当者（a-bud / a-tree / a-leaf 等）
- 技術調査と修正実装
- IC への報告（30 分粒度推奨）

### 3.3 連絡係（Communicator）

- ステークホルダー（経理担当・現場リーダー）への状況伝達
- 該当業務の停止要請 or 一時回避策の周知
- Sev1/2 のみ任命、Sev3 は IC が兼務

### 3.4 記録係（Scribe）

- インシデントタイムライン記録
- Chatwork ログ + 別途 `docs/incidents/inc-YYYYMMDD-HHMM.md` 作成
- Sev1/2 のみ任命

## 4. インシデント対応フロー

### 4.1 全体フロー

```
[1] 検知（監視通知 or 通報）
  ↓
[2] 重大度判定（IC が宣言）
  ↓
[3] チャネル開設（Chatwork ルーム）
  ↓
[4] 役割分担（Sev1/2 のみ Communicator + Scribe 任命）
  ↓
[5] 影響範囲の特定
  ↓
[6] 一時回避策（あれば即適用）
  ↓
[7] 恒久対策（修正実装 / 復元）
  ↓
[8] 終息宣言（IC）
  ↓
[9] ポストモーテム（24-72h 以内）
  ↓
[10] known-pitfalls 追記
```

### 4.2 各 Sev のフロー差分

| ステップ | Sev1 | Sev2 | Sev3 |
|---|---|---|---|
| [3] チャネル開設 | 専用 Chatwork ルーム | 既存ルーム使用可 | 既存ルームで対応 |
| [4] 役割分担 | IC + Responder + Communicator + Scribe | IC + Responder + Scribe | IC のみ |
| [9] ポストモーテム | 必須（24h 以内）| 必須（72h 以内）| 任意 |

## 5. プレイブック（Sev1）

### 5.1 検知 → 5 分以内

```
□ 監視通知（Chatwork）受信
□ Garden トップ画面で再現確認
□ Supabase ダッシュボード Status 確認
□ Vercel ダッシュボード Status 確認
□ /api/health/deep の結果確認
□ 影響範囲の暫定特定
```

### 5.2 5 分 〜 30 分

```
□ Sev1 宣言を Chatwork で発信（テンプレ§5.6）
□ 専用ルーム開設「inc-YYYYMMDD-HHMM-(概要)」
□ Communicator + Scribe 任命
□ ステークホルダーに第一報
□ Garden トップ画面に障害バナー表示
   - 既定: 「現在システム障害が発生しています。対応中です。」
□ 一時回避策の検討（手動運用 / 旧ツールへの一時切替）
```

### 5.3 30 分 〜 1h

```
□ 一時回避策の適用（業務継続を優先）
□ Sev1 → Sev2 降格可否の検討
□ 恒久対策の方針決定
□ ポストモーテム議事録の起票（Scribe）
```

### 5.4 1h 〜 終息

```
□ 恒久対策の実装 + デプロイ
□ 動作確認（再現テスト）
□ ステークホルダーに状況報告（30 分粒度）
□ 障害バナーを「対応中」→「復旧確認中」に更新
```

### 5.5 終息

```
□ 全機能の動作確認完了
□ 障害バナー解除
□ 終息 Chatwork 通知（テンプレ§5.7）
□ ポストモーテム日程の決定（24h 以内）
```

### 5.6 Sev1 第一報テンプレ

```
🚨 Sev1 インシデント発生

発生時刻: 2026-04-26 14:23 JST
症状: 〇〇が動作しない（再現可能）
影響範囲: 全ユーザー / 〇〇モジュール
業務影響: ✅振込実行不可 / ✅給与配信不可 / etc.

IC: 東海林さん
Communicator: 〇〇さん
Scribe: 〇〇さん

対応ルーム: Chatwork "inc-20260426-1423-(概要)"
状況更新: 30 分粒度で本ルームに発信
```

### 5.7 終息テンプレ

```
✅ Sev1 終息

発生: 2026-04-26 14:23 JST
終息: 2026-04-26 15:47 JST
継続時間: 1h24m

原因（暫定）: 〇〇
対応: 〇〇
影響を受けたデータ: なし / N 件（詳細は§後述）

ポストモーテム: 2026-04-27 10:00 開催予定
docs/incidents/inc-20260426-1423.md に記録
```

## 6. プレイブック（Sev2）

### 6.1 検知 → 30 分以内

- 重大度判定 + Chatwork 通知（既存ルーム）
- IC + Responder 着手
- 一時回避策があれば適用

### 6.2 30 分 〜 4h

- 影響特定 + 修正実装 + デプロイ
- 1h 粒度で進捗報告
- 終息見込みが立たない場合 Sev1 への昇格を検討

### 6.3 終息 → 72h

- 終息 Chatwork 通知
- 72h 以内にポストモーテム
- known-pitfalls 追記

---

## 7. プレイブック（Sev3）

- IC が単独対応可
- 通常の修正フロー（PR → レビュー → マージ）で対応
- ポストモーテム任意（記録は GitHub PR で代替可）
- 同種が 3 件以上発生したら Sev2 相当と扱い、ポストモーテム実施

## 8. ポストモーテム

### 8.1 目的

- 原因究明（What happened）
- 影響評価（Impact）
- 改善策抽出（Action items）
- 知見蓄積（known-pitfalls 追記）

### 8.2 開催タイミング

| Sev | 開催期限 | 形式 |
|---|---|---|
| Sev1 | 24h 以内 | 30〜60 分の対面 / Meet |
| Sev2 | 72h 以内 | 30 分以内、対面 or 非同期（Chatwork スレッド可）|
| Sev3 | 任意 | PR 説明欄で代替可 |

### 8.3 ポストモーテム フォーマット

```markdown
# Postmortem - inc-YYYYMMDD-HHMM-(概要)

## メタ情報
- 重大度: Sev1 / Sev2 / Sev3
- 発生: YYYY-MM-DD HH:MM JST
- 終息: YYYY-MM-DD HH:MM JST
- 継続時間: XhXXm
- IC: 〇〇
- Responder: 〇〇
- Scribe: 〇〇

## 影響評価
- ユーザー数: N 名
- 影響業務: 振込実行 / 給与配信 / 〇〇
- データ損失: あり N 件 / なし
- 金銭影響: あり N 円 / なし
- 信頼影響: 経理担当者 N 名から問合せあり

## タイムライン
| 時刻 | イベント | 担当 |
|---|---|---|
| 14:23 | 監視通知 受信 | — |
| 14:25 | 再現確認、Sev1 宣言 | IC |
| 14:27 | Chatwork ルーム開設、第一報 | IC |
| 14:35 | 障害バナー表示 | Responder |
| 14:50 | 原因特定（〇〇） | Responder |
| 15:10 | 一時回避策適用 | Responder |
| 15:40 | 恒久対策デプロイ | Responder |
| 15:47 | 全機能正常確認、終息宣言 | IC |

## 根本原因（5 Whys）
- Why 1: なぜ〇〇が動かなかったか？
  → A1: 〇〇のため
- Why 2: なぜ A1 が起きたか？
  → A2: 〇〇のため
- Why 3: ...
- Why 4: ...
- Why 5: 真因 = 〇〇（プロセス / 仕組み / コード）

## 良かった点（Went well）
- 検知 → IC 宣言まで 5 分以内
- 一時回避策がスムーズに適用された

## 悪かった点（Didn't go well）
- 監視アラートが鳴ってから IC が気付くまで 8 分（夜間）
- ステークホルダー通知が 30 分遅れた

## 改善アクション
| # | アクション | 担当 | 期限 |
|---|---|---|---|
| 1 | 監視通知の DM 連打を critical 即時に変更 | a-main | 2026-04-30 |
| 2 | known-pitfalls.md に〇〇追記 | a-main | 2026-04-28 |
| 3 | テストケース追加（〇〇シナリオ） | a-bud | 2026-05-15 |
| 4 | プレイブックに〇〇手順追加 | a-main | 2026-04-30 |

## 関連リンク
- Chatwork スレッド: https://...
- 修正 PR: https://github.com/...
- 関連監視イベント: monitoring_events.id = ...
```

### 8.4 ポストモーテム原則

- **責任追求ではなく仕組み改善**（Blameless Postmortem）
- 個人名で「〇〇が悪い」と書かない、プロセス・仕組みの改善点として書く
- 改善アクションは**期限と担当を必ず明記**
- 期限超過の改善は IC が再度 Chatwork で確認

## 10. 訓練（Drill）

### 10.1 Sev1 訓練（年 2 回）

- 仮想シナリオ（例：振込画面が全停止）
- IC + Responder + Communicator + Scribe を集めて模擬対応
- 60 分以内に終息宣言を出せるか検証
- 結果を `docs/drills/drill-YYYYMMDD-incident.md` に記録

### 10.2 Sev2 訓練（年 4 回）

- 各モジュール担当が単独で実施
- 監視通知から 4h 以内に終息できるか検証

### 10.3 訓練結果のフィードバック

- プレイブック修正
- known-pitfalls 追記
- 実機材（Chatwork ルーム / バナー機能）の動作確認

## 12. 既知のリスクと対策

### 12.1 IC 不在時の判断遅延

- 対策: 代理 IC ロスター（super_admin 名簿）を `known-pitfalls.md` に常時掲載

### 12.2 Sev 判定の誤り

- 過小評価で Sev3 → 業務影響拡大
- 対策: 影響範囲が不明な場合は **暫定 Sev2** で立ち上げ、5〜10 分で再評価

### 12.3 Chatwork 経由通知の遅延・未達

- Chatwork 自体が落ちる可能性
- 対策: 重要 Sev1 通知は Chatwork + メール（東海林さん個人）の二重経路

### 12.4 ポストモーテム参加者の確保

- 全員集合できないと延期 → 有耶無耶
- 対策: **非同期ポストモーテム**を許容（Chatwork スレッドで 24h 内に意見集約）

### 12.5 改善アクションの放置

- ポストモーテムで決めた改善が実行されない
- 対策: a-main が **週次で改善アクション期限を確認**、超過時 Chatwork で再リマインド
