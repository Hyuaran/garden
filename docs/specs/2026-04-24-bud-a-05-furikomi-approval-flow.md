- 対象: Garden-Bud 振込の承認・差戻し・CSV出力・振込完了 UI
- 見積: **0.5d**（約 4 時間）
- 担当セッション: a-bud
- 作成: 2026-04-24（a-auto / Phase A 先行 batch5 #A-05）
- 前提 spec: Bud A-03（6 段階遷移）, Bud A-04（新規作成）

## 1. 目的とスコープ

### 目的
A-03 の遷移ルールに従って、各ロール（staff / approver / admin / super_admin）が**画面上で実行できるアクション**を UI コンポーネントとして定義する。特に承認待ち画面と詳細画面での遷移ボタンの可視性・動作を明確化。

### 含める
- 振込詳細画面 `/bud/transfers/[id]`
- 承認待ち一覧（`/bud/transfers?status=承認待ち`）での一括操作
- 差戻しモーダル（reason 入力）
- CSV 出力画面（選択一覧 → ZIP DL）
- 振込完了マーク（bud_statements 連携前の暫定モード）
- ステータス変更履歴タブ（bud_transfer_status_history の表示）

### 含めない
- 新規作成フォーム（A-04）
- 明細照合ロジック（A-06）
- CSV フォーマット詳細（別タスク、銀行別）

## 4. データモデル提案

本 spec は UI のため新規テーブル不要。参照：
- `bud_transfers`（既存）
- `bud_transfer_status_history`（A-03 で新規）
- `root_bank_accounts` / `root_vendors`（既存）

### 関連振込タブ用のクエリ
```sql
-- 同一取引先・同時期（過去 12 ヶ月）の振込履歴
SELECT id, transfer_id, amount, scheduled_date, status
  FROM bud_transfers
  WHERE vendor_id = $1
    AND scheduled_date >= CURRENT_DATE - INTERVAL '12 months'
    AND id != $2  -- 自分以外
  ORDER BY scheduled_date DESC
  LIMIT 20;
```

## 5. API / Server Action 契約

### 5.1 既存呼出
- `transitionTransferStatus()` （A-03 §5.1）を approve / reject / mark_csv / mark_completed で利用

### 5.2 新規 Server Action

```typescript
// 一括承認
export async function batchApproveTransfers(params: {
  transferIds: string[];
  reason?: string;         // 通常は不要、必要なら追加可
}): Promise<{
  approved: string[];     // 成功した id
  failed: Array<{ id: string; error: string; code: string }>;
}>;

// 一括差戻し
export async function batchRejectTransfers(params: {
  transferIds: string[];
  reason: string;          // 必須
  notifyChatwork?: boolean;
}): Promise<{
  rejected: string[];
  failed: Array<{ id: string; error: string; code: string }>;
}>;

// CSV 出力（実装の骨格のみ、詳細は別 spec）
export async function exportTransfersCsv(params: {
  transferIds: string[];
  bankGroupBy: 'rakuten' | 'mizuho';  // 銀行別に別 CSV
}): Promise<{
  success: boolean;
  downloadUrl?: string;   // ZIP の signedURL
  filename?: string;
  error?: string;
}>;
```

### 5.3 エラーハンドリング共通
- 一部失敗時は `failed` 配列に列挙、全体成功/失敗フラグは設けない
- UI はユーザーに件数別に通知（例: "2 件承認、1 件失敗"）

## 6. 状態遷移

A-03 §6 の遷移表準拠。本 spec は UI 側で：

### ボタン可視性（role × status 表）
| 画面 | 承認待ち | 承認済み | CSV出力済み | 振込完了 |
|---|---|---|---|---|
| approver 詳細 | ✓承認 / ✗差戻 | - | - | - |
| admin 詳細 | - | 📄 CSV 出力へ | ✅振込完了マーク | - |
| super_admin 詳細 | ✓承認 / ✗差戻 / 📄 CSV / ✅完了 | 同 | 同 | - |
| 起票者（staff）差戻し | - | - | - | - |
| 起票者（staff）差戻し状態 | 下書きに戻す | - | - | - |

### 一括操作
- 承認待ち一覧で**複数選択** → approver が「一括承認」または「一括差戻し」
- CSV 出力済み一覧で**複数選択** → admin が「一括振込完了マーク」（将来、現状は単独）

## 7. Chatwork 通知

### 7.1 承認時
- **通知先**: 起票者（staff）の Chatwork ID
- **内容**: `✓ 振込承認: FK-XXX が承認されました（¥XXX, 取引先: XXX）`
- **トリガ**: `transitionTransferStatus(to='承認済み')` 成功時
- **タイミング**: **即時**（Phase A での簡易実装）

### 7.2 差戻し時
- **通知先**: 起票者
- **内容**: `✗ 振込差戻し: FK-XXX が差戻されました。理由: "{reason}"`
- **トリガ**: 差戻しモーダルで `notifyChatwork=true` チェック時のみ
- **タイミング**: 即時

### 7.3 CSV 出力完了時
- **通知先**: 通知なし（admin が自身で操作、記録は history のみ）

### 7.4 Phase A では日次集約なし
- 即時通知のみ、将来は「承認依頼のまとめ」を日次 09:00 集約（Phase B）

## 9. バリデーション規則

| # | アクション | 事前条件 | 違反時 |
|---|---|---|---|
| V1 | 承認ボタン押下 | status='承認待ち' かつ role='approver'+ | ボタン非活性 |
| V2 | 差戻しボタン押下 | status in ('承認待ち')、reason ≥ 10 文字 | エラー表示 |
| V3 | CSV 出力 | status='承認済み' の振込が 1 件以上 | "対象なし" 表示 |
| V4 | 振込完了マーク | status='CSV出力済み' | ボタン非活性 |
| V5 | 一括承認 | 選択中すべて status='承認待ち' | 不一致分は除外 |
| V6 | 承認者本人の起票は承認不可 | status_changed_by !== 起票者 | 「自己承認は不可」エラー |

V6 は Phase A では**警告のみ**（super_admin 自起票スキップを許容するため）。本運用は Phase B で厳密化判断。

## 10. 受入基準

1. ✅ 振込詳細画面 `/bud/transfers/[id]` が表示され、ステータスに応じたボタンが出る
2. ✅ approver で承認待ち詳細を開く → [承認] [差戻し] の 2 ボタンが出る
3. ✅ 承認クリック → 状態が「承認済み」に遷移、Chatwork 通知が届く
4. ✅ 差戻しモーダルで理由入力 → 10 文字未満で「差戻しを確定」非活性
5. ✅ 承認待ち一覧で複数選択 → 一括承認が動作、成功/失敗件数が表示
6. ✅ admin で承認済み詳細 → [CSV 出力へ] ボタンで出力画面へ遷移
7. ✅ CSV 出力画面で銀行別サマリが正しく表示、ZIP が DL できる
8. ✅ ステータス履歴タブで `bud_transfer_status_history` が降順表示
9. ✅ 起票者本人の承認は警告が出る（V6）
10. ✅ super_admin（東海林さん）は下書き → 承認済みへ直接遷移可（A-03 §6）

## 12. 判断保留

| # | 論点 | a-auto スタンス |
|---|---|---|
| 判1 | 承認権限の「本人承認禁止」を厳密化するか | **警告のみ**で開始、super_admin 自起票スキップと整合させる |
| 判2 | 一括振込完了マーク（複数件）| **Phase A は単一のみ**、Phase B で bud_statements 照合と同時実装 |
| 判3 | 承認・差戻しの comment 欄 | **差戻しのみ必須**、承認時はオプションで追加可（Phase B）|
| 判4 | Chatwork 通知の個人 ID 解決 | `root_employees.chatwork_account_id` カラム追加を前提（別 spec）|
| 判5 | CSV 出力後の「再出力」可否 | 再出力可、history に記録済なら運用上問題なし |
| 判6 | モバイル対応（承認者が出先で承認）| Phase A では PC 前提、モバイル対応は Phase B |
| 判7 | 一括操作の最大件数制限 | **100 件上限**を推奨（UI パフォーマンス）|
