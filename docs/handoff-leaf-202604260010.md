## 修正サマリ

a-review が PR #65 で検出した重大指摘 2 件を修正。

### 脆弱性 1: SECURITY DEFINER 関数で `search_path` 未設定（schema poisoning）

**対象関数 4 つ**すべてに `SET search_path = ''` 追加 + schema 完全修飾:
- `public.leaf_user_in_business(biz_id text)` (§4)
- `public.leaf_kanden_attachments_history_trigger()` (§13)
- `public.verify_image_download_password(input_password text)` (§14)
- `public.set_image_download_password(...)` (§14)

実施した schema 明示:
- `public.<table>` ですべての public テーブル参照を修飾（leaf_user_businesses / root_settings / leaf_kanden_attachments / leaf_kanden_attachments_history）
- `public.is_user_active()` / `public.garden_role_of(...)` で関数呼出も修飾
- `auth.uid()` は `(SELECT auth.uid())` で囲む（防御的、search_path 空でも確実に解決）
- pgcrypto の `crypt()` / `gen_salt()` は `extensions.crypt()` / `extensions.gen_salt()` に明示（Supabase 標準で extensions schema 配下）

これにより、攻撃者が public schema 内に同名の関数/テーブルを作成しても、
SECURITY DEFINER 関数の動作を乗っ取れない。

### 脆弱性 2: クライアントから hash 値を受け取る設計

`set_image_download_password` の引数変更:

**旧**:
```sql
set_image_download_password(new_hash text)
-- クライアントが bcryptjs で hash 化 → サーバへ hash 送信
-- リスク: 任意の hash 値を送信して認証回避可能
```

**新**:
```sql
set_image_download_password(new_password text)
-- クライアントは平文 PW のみ送信
-- サーバ内で extensions.crypt(new_password, extensions.gen_salt('bf', 12)) で hash 化
-- リスク: クライアントから hash を送信できないので認証回避不可
```

`verify_image_download_password` は元々 `input_password text`（平文）を受取り、
サーバ側で `crypt(input_password, stored_hash)` で比較する**正しい設計**のため、
今回は schema 明示のみで機能変更なし。

### §15 初期データ修正

仮 PW `change-me-immediately` の hash 化処理も extensions schema を明示:
```sql
extensions.crypt('change-me-immediately', extensions.gen_salt('bf', 12))
```
