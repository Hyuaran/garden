# 018（私）の仮説：なぜリセットが起きたか

5 つの要因を仮説提示:

- 仮説 1. handoff は「読む」だけで「体内化」されない。MEMORY.md は system reminder で load 済だが、**目を通すこと ≠ 行動を変えること**
- 仮説 2. 起動時 ritual が不在。handoff §11「Step 1-5」は形式的で、ルール体内化を強制していない
- 仮説 3. 「ガンガン本質」と「起動時慎重さ」の優先度衝突。デフォルトがガンガン稼働中 → 起動直後に dispatch 起草・即実行に走る誘因
- 仮説 4. 直前期の重要決定を起動時に強制再確認する装置がない。017 で 50% 使った議論結果が、018 では handoff §7（memory 棚卸し結果）の項目名一覧で済まされる
- 仮説 5. 「違反集計（§6）の再発防止内面化」を起動時にチェックする仕組みがない

# 018（私）の対策案：6 案、まだ推奨絞りこめていない

- 案 A. handoff に「a-main-N+1 必須実施 起動時チェックリスト」テンプレ埋め込み + 新セッション側で全項目 ☑ 完了前に dispatch 起草を**ロック**
- 案 B. handoff §A として「前期で議論した重要決定 N 件 summary + 再発防止策 + 体内化確認方法」を必須セクション化、起動時に Read + 確認返答強制
- 案 C. memory 「最重要 9 件」を起動時に 1 件ずつ Read → 要約返答する強制ステップ
- 案 D. 応答前 sentinel に「直前期の主要議論体内化チェック」追加、違反検出で自動一時停止
- 案 E. 起動時チェックリスト未完了の場合「dispatch 起草・即実行禁止」のロック（A と統合可能）
- 案 F. 017 への dispatch で自己分析依頼（本 dispatch がそれ）
