## 概要

給与を **銀行振込ではなく現金手渡し** で受取る従業員の業務フローを Phase B 着手前に確定させたい。

**5 論点（採択判断）+ 5 未確認事項（情報収集）= 10 項目** を以下の表形式で東海林さんから即決いただきたい。

---

## 5 論点（採択判断）

### 論点 1: 手渡し受給者の識別方法

| 案 | 内容 | a-auto 推奨 |
|---|---|---|
| A | `root_employees.payment_method` カラム追加（ENUM: bank_transfer / cash / other） | **★ 推奨** — マスタ集約、MF クラウド給与連携でも参照可 |
| B | `bud_salary_config` テーブル新設 | Bud 内完結、参照粒度が限定 |
| C | `bud_transfers.transfer_type='給与(手渡し)'` のみ | 設定情報が振込レコードにしか残らない |

**東海林さん回答欄**: A / B / C ＿＿

---

### 論点 2: `bud_transfers` への扱い

| 案 | 内容 | a-auto 推奨 |
|---|---|---|
| A | 登録しない（現金は振込ではない） | データ単純、ただし集計漏れリスク |
| B | 種別付きで登録（`transfer_type='給与(手渡し)'`、CSV 出力対象から除外） | **★ 推奨** — 集計一元化、明細配信も統一 |
| C | 別テーブル `bud_cash_payments` | 完全分離、JOIN 増 |

**東海林さん回答欄**: A / B / C ＿＿

---

### 論点 3: 給与明細の配信（Garden-Tree 側）

| 案 | 内容 | a-auto 推奨 |
|---|---|---|
| A | 全員 Garden-Tree マイページで明細閲覧（手渡しも含む） | **★ 推奨** — 配信元一本化、UX 統一 |
| B | 手渡し受給者のみ別ルートで PDF メール | 二重管理 |
| C | 手渡し受給者は明細配信なし（紙で同梱） | 紙印刷コスト |

**東海林さん回答欄**: A / B / C ＿＿

---

### 論点 4: 現金原資の管理

| 案 | 内容 | a-auto 推奨 |
|---|---|---|
| A | ATM 引出しを `bud_statements` 出金として記録、費目 '現金原資' | 透明性高 |
| B | 現金原資は経理担当の手元現金として Bud 外管理 | 監査追跡困難 |
| C | 月初に予算化、実績と突合 | 予算画面が必要 |
| **A+C 併用** | 両方の利点を取る | **★ 推奨** — Phase B で予算画面追加 |

**東海林さん回答欄**: A / B / C / A+C ＿＿

---

### 論点 5: 受領確認

| 案 | 内容 | a-auto 推奨 |
|---|---|---|
| A | 紙の受領書（押印 or 署名）を徴求 + スキャン保管 | 監査対応（物理証拠） |

**東海林さん回答欄**: A / B / C / A+B ＿＿

### 論点 3 の拡張採択（重要）

東海林さんから **A 案推奨 + メール配信追加** で採択。詳細：

- **配信方式**: **方式 2 パスワード保護 PDF 添付**（メール本文に hint 文同梱）
- **配信先**: マイページに登録されているメールアドレス
- **PW**: 生年月日 4 桁（MMDD）or 社員番号下 4 桁（**実装時に最終決定**）
- **本文 hint**: 「PW は生年月日 4 桁です」等を明示
- **重要前提**: Garden ログイン = 社内 PC 限定（通常ロール）→ メール配信が**自宅確認の唯一経路**
- **Garden ログイン社内 PC 限定**は Root Phase B-5（認証セキュリティ強化）で実装
