## 5. 🟢 楽天独自パーサーの実装方針（確定）

### 5.1 列定義（実 CSV に基づく）

```typescript
// src/lib/shiwakechou/parsers/bank/rakuten.ts
const RAKUTEN_HEADER = "取引日,入出金(円),残高(円),入出金先内容";

type RakutenTransactionRow = {
  transaction_date: Date;     // YYYYMMDD → Date
  amount: number;             // 負=出金、正=入金 (絶対値で bud_transactions.amount)
  flow: "withdrawal" | "deposit";  // amount 符号から導出
  balance: number;            // 取引後残高（円）
  description: string;        // 入出金先内容（摘要）
};
```

### 5.2 解析手順

1. **Buffer → Shift-JIS デコード**: `iconv-lite`（既に Bud Phase 1a で導入済 npm パッケージ、流用可）
2. **CRLF 分割 + 空行除去**
3. **ヘッダー行検証**: lines[0] === RAKUTEN_HEADER（厳格マッチ、不一致は警告 + import 中止）
4. **データ行 4 列パース**:
   - `transaction_date`: YYYYMMDD → ISO date
   - `amount`: parseInt、負/正で flow 導出、絶対値を bud_transactions.amount に
   - `balance`: parseInt
   - `description`: そのまま（半角カナ + 全角混在 / 100+ 文字あり）
5. **bud_transactions に bulk insert**（UNIQUE 制約による重複検出）

### 5.3 特殊行への対応（pre-check で発見）

**振込手数料行**（末尾 2 行目に発見）：
```
20260430,-229,22118184,振込手数料（内消費税 ￥20）
```
- 取引内容 = `振込手数料（内消費税 ￥20）`、金額 = -229 円
- **振込手数料は独立した取引行** として bud_transactions に格納（楽天では個別行として出力）
- 共通マスタで「振込手数料」パターンに該当 → 借方=支払手数料、貸方=普通預金、税区分=対象外（手数料の消費税は内訳として既に提示）

**既存 Python の処理**: `1_共通マスタ_v12.xlsx` に「振込手数料」マスタエントリがあるはず（5/6 朝の Excel 読込時に確認）。

### 5.4 description フィールドの扱い

- 最大 100+ 文字あり、半角カナ + 全角混在
- bud_transactions.description は `text` 型（無制限）→ 問題なし
- 共通マスタ判定ロジックは **半角カナ + 全角の正規化** が必要（Python ロジック流用、ロジック L31 注記参照）
