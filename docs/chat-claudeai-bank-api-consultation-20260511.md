# 件名
Garden プロジェクト — Bud 経理モジュールの口座残高表示について、現状の CSV インポート方式を API 連携に置き換える可能性を検討したい。4 銀行（楽天 / みずほ / PayPay / 京都）の API 仕様確認 + 統合方針提案を相談したい。

# プロジェクト背景

Garden（自社 Web アプリ）の経理モジュール「Bud」では、6 法人 × 4 銀行（楽天 / みずほ / PayPay / 京都）の口座残高を一覧表示する画面を実装中です。

経緯:
- 元々マネーフォワードクラウド会計（MFC 会計）を利用していた
- 現在、何らかの理由で MFC 会計の継続利用が困難な状況
- 代替として Garden Bud モジュールで独自実装を進めている
- 既存 MFC 給与 CSV 仕様は別途 Bud Phase D で踏襲中（給与計算系、本相談とは別議題）

現状仕様（CSV インポート方式）:
- 各銀行 net banking から CSV ダウンロード → 手動 or 自動アップロード → Bud DB に取込
- DB テーブル: bud_bank_accounts（口座マスタ）/ bud_bank_balances（残高履歴）/ bud_bank_transactions（取引明細）
- 既に実装済（PR #159 で alpha 完成、5/11 merged）
- 課題: 手動 DL のオペレーション負荷、リアルタイム性なし、MFC 会計の代替として運用負荷が高い

検討したいこと（API 連携への移行）:
- CSV を経由せず銀行 API で残高 + 取引明細を直接取得
- リアルタイム同期（または定期 polling）
- 4 銀行ごとに API 仕様が異なる前提 → 統合実装方針も検討
- MFC 会計の代替として、独自実装でも MFC 同等以上の操作性 / リアルタイム性を確保したい

# 相談したいこと

## 1. 各銀行 API の主要仕様確認

各銀行の API ドキュメント（上記 URL）を読んで、以下を整理してほしい:

| 観点 | 確認したい内容 |
|---|---|
| 認証方式 | OAuth 2.0 / API キー / JWT / その他、契約手順含む |
| 取得可能データ | 残高 / 取引明細 / 振込履歴 / 口座マスタ 等 |
| 法人契約必須か | 個人事業主可 / 法人専用 / 開発者登録要否 |
| 料金体系 | 初期費用 / 月額費用 / API コール課金 |
| レート制限 | 1 秒 / 1 分 / 1 時間 / 1 日 あたりのコール上限 |
| 開発者登録手順 | ドキュメント / sandbox 環境 / 本番申請フロー |
| 想定実装言語 / SDK | REST / GraphQL / 公式 Node.js SDK 等 |
| セキュリティ要件 | TLS / IP allowlist / signing 等 |

## 2. 京都銀行 API の代替検索

京都銀行の API ドキュメントが見つからない場合、以下を調査:
- 京都銀行が法人向け口座管理 API を提供しているか
- 提供していない場合、代替方法（CSV 継続 / Web スクレイピング / 他サービス連携）
- 経済産業省 / 金融庁 / 銀行間 API 標準（オープン API）への準拠状況

## 3. Garden 統合実装の推奨方針

4 銀行を Garden（Next.js + Supabase + TypeScript）に統合する場合の推奨アーキテクチャ:

| 候補 | 内容 |

| B | それぞれ個別 client、共通化なし |
| C | 段階導入: 楽天 + PayPay 先行（API 仕様が明確そう）、みずほ + 京都は CSV 継続 |
| D | 全銀行 CSV 継続、API 連携は将来検討（5/13 仕訳帳本番運用優先）|

各候補のメリット / デメリット / 推奨度（5/13 本番運用直前の現状で）
